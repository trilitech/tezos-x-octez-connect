POC In progress Tezos X

Beacon Multi-chain — Development Plan

Proof of concept for extending Beacon / Octez.connect to support single-session multi-chain operations across Tezos L1 and the Michelson interface of Tezos X. Covers three transports: PostMessage, Matrix P2P, and WalletConnect v2.

April 2026 · Tezos X Adoption Team

Context

Beacon (TZIP-10) today supports a single network per session. Tezos X introduces a second runtime — the Michelson interface — that shares the same account model as Tezos L1 but lives on a separate chain. A dApp targeting Tezos X needs to operate on both chains within a single wallet session.

This POC validates the required protocol changes to Beacon (TZIP-10) and the wallet-side implementation across all three Beacon transports, on a real dual-runtime previewnet.

The SDK fork (@airgap/beacon-sdk) used throughout the POC serves as a reference implementation for the TZIP proposal. At POC completion, changes are distilled into a TZIP document and a PR against the upstream SDK. The fork is then archived.

Objectives
Objective 1

UX demonstration

Show what single-session multi-chain UX looks like for a user: one pairing, one approval screen, operations on both L1 and the Michelson interface without re-connecting.

Objective 2

Protocol change demonstration

Show the concrete delta to the Beacon/TZIP-10 protocol — a new version_upgrade_request message type for capability negotiation, the networks array on permission_request v3, explicit network routing on operation_request — and prove backward compatibility.

Objective 3

Transport coverage

Validate multi-chain sessions across all three Beacon transports (Matrix P2P, WalletConnect v2, PostMessage popup) and in a real cross-domain deployment — confirming the protocol extension is transport-agnostic.

Test infrastructure

Headless wallet + agent test runner

Each phase must be validatable by an automated agent, not just a human. The wallet exposes a headless mode and a local HTTP control API; a Node.js test runner drives the full flow without UI interaction.

Headless mode (HEADLESS=1) — both wallet and dApp

  • Launched with env flag or URL param: ?headless=1 — dev/test only, never built into production
  • Wallet: auto-approves all permission_request and operation_request; auto-responds to version_upgrade_request with version_upgrade_response when HEADLESS=1
  • dApp exposes: POST /request-permissions — comportement selon le body : sans networks (body vide ou {}) → flow v2 seulement, envoie uniquement permission_request v2, saute les étapes 2 et 3 (utilisé en Phase 1) ; avec { networks: [...] } → séquence 3 étapes complète (permission_request v2 → version_upgrade_request → permission_request v3 + networks[]) ; si le wallet ne répond pas à l'upgrade dans les 5s, reste en mode legacy v2 (utilisé en Phase 2+). URI disponible immédiatement après l'appel ; GET /pairing-uri — returns the URI generated by the last /request-permissions call ; POST /request-operation — triggers dAppClient.requestOperation({ network?, operationDetails }), returns { transactionHash } ; GET /last-op — last { transactionHash } ; GET /last-permission — last permission response: { version: "3", accounts: { chainId: { publicKey } } } in v3 mode, { version: "2", publicKey } in v2 mode ; GET /last-handshake — handshake trace: { step1: "acked", step2: "acked"|"timeout", step3: "acked"|null, mode: "v2"|"v3" } — lets tests verify the correct code path was taken, not just the outcome ; POST /reset
  • Wallet exposes: POST /connect — accepts a dApp pairing URI and calls WalletClient.addPeer() programmatically; GET /last-rpc-call — returns { chainId, rpcUrl } for the last operation_request routed (validates actual HTTP dispatch, not just log); POST /reset
  • Prerequisite spike before Phase 1 (≤ 2h): read @airgap/beacon-sdk source and answer two questions: (1) confirm pairing flow direction — assumption: dApp generates URI, wallet calls addPeer(); verify against SDK source before building; (2) TypeScript API surface — what changes are needed to approvePermissionRequest and PermissionResponse to support per-chain accounts? Also confirm that unpatched SDK ignores unknown message types (like version_upgrade_request) gracefully — no exception thrown that would break the session. Update headless API design and Phase 2 protocol spec accordingly. This spike is a hard gate — Phase 1 cannot start without it.

Agent test runner (test/run.ts)

  • Phases 1–3, 5: HTTP-based — drives dApp and wallet via headless API endpoints. Phase 4 exception: HTTP-based + one lightweight Playwright step for CORS detection (CORS errors are browser-side; a Node.js test runner cannot observe them).
  • All URLs parameterised via env vars: DAPP_URL (default localhost:5173), WALLET_URL (default localhost:5174) — Phase 4 overrides these with deployed domains
  • Phase 6 exception: Playwright full-browser driver — wallet runs as a popup inside the browser, no HTTP server. Auto-approve via ?headless=1 URL param. Playwright controls both dApp page and popup window directly.

// test/phase1.ts — agent-runnable pattern (HTTP-based, no SDK access)
// Pairing flow direction: dApp generates URI → test runner fetches → passes to wallet via POST /connect.
// Assumption confirmed in prerequisite spike (read beacon-sdk source, verify addPeer() call site).

// 1. Trigger permissions (generates pairing URI on dApp side)
await fetch(`${DAPP_URL}/request-permissions`, { method: 'POST' });
// 2. Fetch URI and pass to wallet
const uri = await fetch(`${DAPP_URL}/pairing-uri`).then(r => r.text());
await fetch(`${WALLET_URL}/connect`, { method: 'POST', body: uri }); // wallet auto-approves
// 3. Trigger operation, get hash
const { transactionHash: hash } = await fetch(`${DAPP_URL}/request-operation`, {
  method: 'POST', body: JSON.stringify({ operationDetails: [{ kind: 'transaction', amount: '1', destination: DEST }] })
}).then(r => r.json());
// 4. Confirm on-chain
await waitForConfirmation(hash, L1_RPC);
const allOps = await Promise.all([0,1,2,3].map(i =>
  fetch(`${L1_RPC}/chains/main/blocks/head/operations/${i}`).then(r => r.json())
));
const op = allOps.flat().find(o => o.hash === hash); // all 4 passes — avoids false negative on reveal or other non-manager ops
assert(op !== undefined, 'operation not found on chain');

Shared helper: waitForConfirmation(hash, rpcUrl)

  • Polls GET /chains/main/blocks/head/operations/${i} for passes 0–3 every 2s (same pattern as the Phase 1 assertion) — flattens all 4 pass arrays with .flat().find(o => o.hash === hash); resolves when the operation is found
  • Timeout after 60s → test fails
  • Returns block level on success

const signer = await InMemorySigner.fromSecretKey('edsk...');
const Tezos = new TezosToolkit('https://demo.txpark.nomadic-labs.com/rpc/tezlink');

TZIP-10 protocol extension

Nouveau type version_upgrade_request, networks[] sur permission_request v3, routage sur operation_request
Matrix P2P TZIP-10 delta
Not started

Core protocol change. Extend the Beacon message format, implement on both sides, validate backward compatibility with an unpatched wallet. Both @airgap/beacon-dapp and @airgap/beacon-wallet need changes — the POC works from a fork of @airgap/beacon-sdk; monkey-patching a complex SDK is too fragile for multi-phase work.

Phase 2 deliverables beyond the protocol: (1) Fake Michelson network points to a distinct local proxy (http://localhost:9998/tezlink) that logs requests and forwards to L1 RPC — routing validated at HTTP dispatch level, not just log. (2) Second headless wallet build using stock (unpatched) @airgap/beacon-wallet served at WALLET_UNPATCHED_URL for backward compat test. (3) HEADLESS_APPROVE_NETWORKS=<chainId> flag to limit which networks the wallet approves — enables subset approval test.

Protocol delta

  • Version negotiation en 3 étapes (aucune ambiguïté de timeout) :
    1. dApp envoie permission_request version: "2" standard → session établie, wallet confirmé vivant
    2. dApp envoie version_upgrade_request (nouveau type) avec targetVersion: "3" → timeout court ~5s. Si réponse reçue : wallet patché, v3 actif. Si timeout : wallet v2 (on sait qu'il est en ligne, donc timeout = pas de support v3) → mode legacy.
    3. Si v3 confirmé : dApp envoie permission_request version: "3" + networks[] → wallet répond avec comptes indexés par chain. Sémantique du second permission_request : un second permission_request sur une session existante remplace la session précédente — le wallet écrase les permissions stockées. Ce comportement doit être implémenté explicitement dans le fork SDK (ne pas rejeter la requête comme erreur de protocole). Si l'étape 3 ne reçoit pas de réponse, le dApp tombe en erreur (pas de fallback silencieux — l'upgrade a déjà été confirmé à l'étape 2).
    Le round-trip supplémentaire (étapes 1→2) est intentionnel pour cette version POC et pourra être supprimé en production une fois le support v3 largement déployé.
  • Add networks: NetworkInfo[] to permission_request v3 — chaque entry contient chainId + rpcUrl. Coexistence network / networks[] : le champ network (singulier) est conservé en v3 uniquement pour compatibilité avec le parsing des vieux SDK — il ne définit pas de "réseau primaire". Un wallet v3 qui reçoit les deux champs utilise networks[] et ignore network. Cette règle doit figurer explicitement dans la TZIP.
  • Add network: string (bare CAIP-2 chain ID, e.g. "tezos:NetXH12Aer3be93") to operation_request — required in multi-chain mode (v3), absent in legacy mode (v2 fallback). Note de schéma : contrairement au champ network de permission_request (objet { type, chainId?, rpcUrl? }), le champ network de operation_request est un string plat — seul le chain ID est nécessaire pour le routing, pas les métadonnées réseau. Cette divergence de shape doit être documentée explicitement dans la TZIP.
  • No "primary network" concept — two clean modes, explicit version-negotiated
  • SDK type changes required (confirmed in spike): PermissionResponse must gain an accounts: Record<chainId, { publicKey }> field; approvePermissionRequest(request, { publicKey }) must be extended to accept per-chain accounts. The wallet builds a RPC URL registry keyed by chain ID from permission_request.networks[].rpcUrl — used to route subsequent operation_request messages. Trust model caveat (hors scope POC) : le wallet fait confiance à la dApp pour les URLs RPC — un nœud malveillant fourni par la dApp peut observer ou altérer les opérations. À adresser dans la TZIP : recommander que les wallets ignorent rpcUrl en faveur de leur propre nœud configuré, ou valident l'URL contre une liste connue.

Derisking questions

  • Backward compat — un wallet non-patché ne répond pas au version_upgrade_request (type inconnu, ignoré) → timeout 5s → dApp reste en mode legacy v2. L'ambiguïté "offline ou rejet ?" est levée : le wallet vient de répondre à la permission_request v2 à l'étape 1, donc on sait qu'il est en ligne. Caveat: validé uniquement contre le SDK headless à WALLET_UNPATCHED_URL — les wallets production (Kukai, Temple) sont hors scope pour ce POC.
  • What happens if the wallet approves only a subset of requested networks?
  • Is the network field on operation_request sufficient or does the dApp need a separate per-network client?
  • Session lifecycle: what happens if the user disconnects mid-session, or revokes access to one chain only? Beacon disconnect tears down the entire session — partial revoke is undefined in current TZIP-10. Out of scope for this POC but must be flagged in the TZIP proposal.

WalletConnect v2 transport

Multi-chain WC2 session with two tezos: chains in a single session proposal
WalletConnect v2 CAIP-25
Not started
Partial blocker: Steps 1–3 (WC2 session setup, namespace negotiation, L1 operation) can be validated with the fake chain ID — the Reown relay is content-blind. Step 4 (Michelson on-chain confirmation) requires the real Michelson RPC endpoint and distinct chain ID — same infrastructure dependency as Phase 3.

WalletConnect v2 targets wallets like Kukai and Bento that support WC2. Changes are localized to WalletConnectCommunicationClient in Octez.connect (separate repo, external dependency) — this work happens in a branch of Octez.connect in parallel; the tezos-x-beacon repo documents the protocol design and provides the E2E test suite. The TZIP-10 protocol changes from Phase 2 carry over unchanged; only the transport layer differs. This phase validates that the WC2 relay handles a session with two tezos: chains and that multi-chain routing translates correctly to CAIP-27 chainId routing.

Tasks

  • Patch WalletConnectCommunicationClient to include both chains in session proposal
  • Extend wallet to handle WC2 session with two tezos: namespaces
  • Map operation_request.network → wc_sessionRequest.chainId
  • Extend headless mode to auto-approve WC2 session proposals (in addition to Beacon messages) — the wallet must call signClient.approve() automatically when HEADLESS=1. State-machine coordination required: WC2 proposals arrive asynchronously via signClient.on('session_proposal', ...); the wallet must initialize the WC2 SignClient, connect the relay WebSocket, and register the event listener before the test sends signClient.connect(). Solution: wallet exposes POST /wc2-ready that returns 200 only once the SignClient is initialized and listening — the test polls this endpoint (with timeout) before proceeding.
  • Validate with Reown relay — Project ID: fb4d4407a8fe167d79bd14b5afcc7230

Derisking questions

  • Design decision — gate avant Phase 5 : le flow 3 étapes (permission v2 → version_upgrade_request → permission v3) s'applique à Matrix et PostMessage parce qu'une session est déjà établie avant l'upgrade. En WC2 il n'y a pas de "session v2 d'abord" — la session proposal WC2 est le premier message, et les namespaces sont négociés atomiquement. Deux options à décider :
    • Option A — WC2 natif : session proposal inclut L1 en requiredNamespaces et Michelson en optionalNamespaces. Si le wallet approuve Michelson → session multi-chain établie. Mapping du handshake Beacon 3 étapes sur WC2 : l'étape 1 (permission_request v2) est remplacée par la WC2 session proposal — c'est elle qui confirme la liveness du wallet. Les étapes 2 (version_upgrade_request) et 3 (permission_request v3 + networks[]) transitent dans le canal WC2 établi après session. /last-handshake en WC2 retourne donc { step1: "wc2-session", step2: "acked"|"timeout", step3: "acked"|null, mode: "v2"|"v3" }. Attention : un wallet v3 partial qui approuve seulement L1 donne une session WC2 valid avec length-1 — le dApp doit tester chains.includes(MICHELSON) pour détecter la capacité.
    • Option B — WC2 opaque : session proposal avec L1 seulement, Beacon v3 en blob opaque dans les messages WC2. Problème : signClient.request({ chainId: MICHELSON_CHAIN_ID }) ne peut pas router vers Michelson si ce chain n'est pas dans session.namespaces. Cette option casse le routing WC2 pour les opérations Michelson. Non viable.

Beacon has a PostMessage transport for browser extensions. This phase adapts or extends it to support a popup model: the dApp opens the wallet via window.open(); because the opener holds a reference to the popup's window object, PostMessage works between the two pages. If the Beacon SDK already supports popup initialization, this is a small extension; if not, a new transport variant is implemented from scratch in the SDK fork.

Test architecture change: the wallet runs as a popup inside a browser — no standalone HTTP server, so the headless HTTP API (Phases 1–5) doesn't apply. Instead, the wallet auto-approves via ?headless=1 URL param, and Playwright controls both the dApp page and the popup window directly. No POST /connect needed — the dApp opens the wallet itself via window.open().
