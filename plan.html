POC In progress Tezos X

Beacon Multi-chain — Development Plan

Proof of concept for extending Beacon / Octez.connect to support single-session multi-chain operations across Tezos L1 and the Michelson interface of Tezos X. Covers three transports: PostMessage, Matrix P2P, and WalletConnect v2.

April 2026 · Tezos X Adoption Team

Context

Beacon (TZIP-10) today supports a single network per session. Tezos X introduces a second runtime — the Michelson interface — that shares the same account model as Tezos L1 but lives on a separate chain. A dApp targeting Tezos X needs to operate on both chains within a single wallet session.

This POC validates the required protocol changes to Beacon (TZIP-10) and the wallet-side implementation across all three Beacon transports, on a real dual-runtime previewnet.

The SDK fork (@airgap/beacon-sdk) used throughout the POC serves as a reference implementation for the TZIP proposal. At POC completion, changes are distilled into a TZIP document and a PR against the upstream SDK. The fork is then archived.

Objectives
Objective 1

UX demonstration

Show what single-session multi-chain UX looks like for a user: one pairing, one approval screen, operations on both L1 and the Michelson interface without re-connecting.

Objective 2

Protocol change demonstration

Show the concrete delta to the Beacon/TZIP-10 protocol — the networks array on permission_request, explicit network routing on operation_request — and prove backward compatibility.

Objective 3

Transport coverage

Validate multi-chain sessions across all three Beacon transports (Matrix P2P, WalletConnect v2, PostMessage popup) and in a real cross-domain deployment — confirming the protocol extension is transport-agnostic.

Test infrastructure

Headless wallet + agent test runner

Each phase must be validatable by an automated agent, not just a human. The wallet exposes a headless mode and a local HTTP control API; a Node.js test runner drives the full flow without UI interaction.

Headless mode (HEADLESS=1) — both wallet and dApp

  • Launched with env flag or URL param: ?headless=1 — dev/test only, never built into production
  • Wallet: auto-approves all permission_request and operation_request
  • dApp exposes: POST /request-permissions — triggers dAppClient.requestPermissions({ networks: [...] }) with body params, URI becomes available immediately after; GET /pairing-uri — returns the URI generated by the last /request-permissions call; POST /request-operation — triggers dAppClient.requestOperation({ network?, operationDetails }), returns { transactionHash }; GET /last-op — last transactionHash; POST /reset
  • Wallet exposes: POST /connect — accepts a dApp pairing URI and calls WalletClient.addPeer() programmatically; GET /last-rpc-call — returns { chainId, rpcUrl } for the last operation_request routed (validates actual HTTP dispatch, not just log); POST /reset
  • Prerequisite spike before Phase 1 (≤ 2h): read @airgap/beacon-sdk source and answer three questions that gate later phases: (1) pairing flow direction — who generates the URI, who calls addPeer(); (2) version envelope handling — does an unpatched SDK reject or silently ignore a permission_request with version: "3"? (if it rejects at the envelope level, no response arrives and the v2 fallback never triggers — this gates the entire capability detection design in Phase 2); (3) TypeScript API surface — what changes are needed to approvePermissionRequest and PermissionResponse to support per-chain accounts? Update headless API design and Phase 2 protocol spec accordingly. This spike is a hard gate — Phase 1 cannot start without it.

Agent test runner (test/run.ts)

  • Phases 1–5: HTTP-based — drives dApp and wallet via headless API endpoints
  • All URLs parameterised via env vars: DAPP_URL (default localhost:5173), WALLET_URL (default localhost:5174) — Phase 4 overrides these with deployed domains
  • Phase 6 exception: Playwright full-browser driver — wallet runs as a popup inside the browser, no HTTP server. Auto-approve via ?headless=1 URL param. Playwright controls both dApp page and popup window directly.
  • Each phase has its own test file: test/phase1.ts … test/phase6.ts
  • Exit 0 = pass, exit 1 + diff = fail
// test/phase1.ts — agent-runnable pattern (HTTP-based, no SDK access)
// Pairing flow direction confirmed in prerequisite spike before Phase 1.
// Assumed: dApp generates URI → test runner fetches → passes to wallet via POST /connect.

// 1. Trigger permissions (generates pairing URI on dApp side)
await fetch(`${DAPP_URL}/request-permissions`, { method: 'POST' });
// 2. Fetch URI and pass to wallet
const uri = await fetch(`${DAPP_URL}/pairing-uri`).then(r => r.text());
await fetch(`${WALLET_URL}/connect`, { method: 'POST', body: uri }); // wallet auto-approves
// 3. Trigger operation, get hash
const { transactionHash: hash } = await fetch(`${DAPP_URL}/request-operation`, {
  method: 'POST', body: JSON.stringify({ operationDetails: [{ kind: 'transaction', amount: '1', destination: DEST }] })
}).then(r => r.json());
// 4. Confirm on-chain
await waitForConfirmation(hash, L1_RPC);
const op = await fetch(`${L1_RPC}/chains/main/blocks/head/operations/3`)
             .then(r => r.json()).then(ops => ops.find(o => o.hash === hash));
assert(op !== undefined, 'operation not found on chain');

Shared helper: waitForConfirmation(hash, rpcUrl)

  • Polls GET /chains/main/blocks/head/operation_hashes every 2s — response is a 4-element nested array (one per validation pass); helper flattens with .flat().includes(hash) before resolving
  • Timeout after 60s → test fails
  • Returns block level on success

// wallet — core pattern
const signer = await InMemorySigner.fromSecretKey('edsk...');
const Tezos = new TezosToolkit('https://demo.txpark.nomadic-labs.com/rpc/tezlink');
Tezos.setSignerProvider(signer);

const client = new WalletClient({ name: "Tezos X Wallet POC" });
await client.init();

client.onPermissionRequest(async (request) => {
  await client.approvePermissionRequest(request, { publicKey: await signer.publicKey() });
});

client.onOperationRequest(async (request) => {
  // Forge + sign + inject via Taquito, return hash
  const op = await Tezos.contract.batch(request.operationDetails).send();
  await client.approveOperationRequest(request, { transactionHash: op.hash });
});

TZIP-10 protocol extension

Add networks[] to permission_request, network routing to operation_request
Matrix P2P TZIP-10 delta
Not started

Core protocol change. Extend the Beacon message format, implement on both sides, validate backward compatibility with an unpatched wallet. Both @airgap/beacon-dapp and @airgap/beacon-wallet need changes — the POC works from a fork of @airgap/beacon-sdk; monkey-patching a complex SDK is too fragile for multi-phase work.

Phase 2 deliverables beyond the protocol: (1) Fake Michelson network points to a distinct local proxy (http://localhost:9998/tezlink) that logs requests and forwards to L1 RPC — routing validated at HTTP dispatch level, not just log. (2) Second headless wallet build using stock (unpatched) @airgap/beacon-wallet served at WALLET_UNPATCHED_URL for backward compat test. (3) HEADLESS_APPROVE_NETWORKS=<chainId> flag to limit which networks the wallet approves — enables subset approval test.

Protocol delta

  • Add networks: NetworkInfo[] to permission_request
  • Wallet responds with accounts indexed per network in permission_response
  • Version negotiation with fallback: dApp sends version: "3" in permission_request. If wallet responds version: "3" → multi-chain mode active. If wallet responds version: "2" or errors → dApp falls back to single-network legacy mode transparently. Version bump is unavoidable for any production protocol change — reusing it for negotiation adds no extra complexity. ⚠ Risk confirmed in spike: if an unpatched SDK rejects version: "3" at the envelope level (before reading fields), no response arrives and fallback never triggers — spike finding (question 2) determines whether this mechanism is viable at all.
  • Add network: string to operation_request — required in multi-chain mode (v3), absent in legacy mode (v2 fallback)
  • No "primary network" concept — two clean modes, explicit version-negotiated
  • SDK type changes required (confirmed in spike): PermissionResponse must gain an accounts: Record<chainId, { publicKey }> field; approvePermissionRequest(request, { publicKey }) must be extended to accept per-chain accounts. The wallet builds a RPC URL registry keyed by chain ID from permission_request.networks[].rpcUrl — used to route subsequent operation_request messages.

Derisking questions

  • Backward compat (unpatched wallet ignores networks) — guaranteed by Beacon protocol design: unknown fields must be ignored by conformant implementations. Caveat: validated only against the SDK's own headless WalletClient at WALLET_UNPATCHED_URL — production wallets (Kukai, Temple) may handle the version field differently and are out of scope for this POC.
  • What happens if the wallet approves only a subset of requested networks?
  • Is the network field on operation_request sufficient or does the dApp need a separate per-network client?
  • Session lifecycle: what happens if the user disconnects mid-session, or revokes access to one chain only? Beacon disconnect tears down the entire session — partial revoke is undefined in current TZIP-10. Out of scope for this POC but must be flagged in the TZIP proposal.
// permission_request — v3 extended format
{
  type: "permission_request",
  version: "3",                          // bumped: signals multi-chain capability
  network: { type: "custom", chainId: "tezos:NetXH12Aer3be93" }, // kept for backward compat
  networks: [                              // new field
    { type: "custom", name: "Tezos L1", chainId: "tezos:NetXH12Aer3be93",
      rpcUrl: "https://demo.txpark.nomadic-labs.com/rpc/tezlink" },
    { type: "custom", name: "Michelson interface",
      rpcUrl: "https://demo.txpark.nomadic-labs.com/rpc/tezlink",
      chainId: "<MICHELSON_CHAIN_ID>" }
  ],
  scopes: ["operation_request"]
}

// permission_response — dApp reads version to detect mode
// version "3" → multi-chain active; version "2" → legacy wallet, fallback
{ version: "3", accounts: { "tezos:NetXH12Aer3be93": { publicKey: "..." }, "tezos:<MICHELSON_CHAIN_ID>": { publicKey: "..." } } }

// operation_request — network required in multi-chain mode
await dAppClient.requestOperation({
  network: "<MICHELSON_CHAIN_ID>",   // required when session is multi-chain (v3)
  operationDetails: [...]
});

// legacy single-network — unchanged, network field absent
await dAppClient.requestOperation({ operationDetails: [...] });

WalletConnect v2 transport

Multi-chain WC2 session with two tezos: chains in a single session proposal
WalletConnect v2 CAIP-25
Not started
Partial blocker: Steps 1–3 (WC2 session setup, namespace negotiation, L1 operation) can be validated with the fake chain ID — the Reown relay is content-blind. Step 4 (Michelson on-chain confirmation) requires the real Michelson RPC endpoint and distinct chain ID — same infrastructure dependency as Phase 3.

WalletConnect v2 targets wallets like Kukai and Bento that support WC2. Changes are localized to WalletConnectCommunicationClient in Octez.connect (separate repo, external dependency) — this work happens in a branch of Octez.connect in parallel; the tezos-x-beacon repo documents the protocol design and provides the E2E test suite. The TZIP-10 protocol changes from Phase 2 carry over unchanged; only the transport layer differs. This phase validates that the WC2 relay handles a session with two tezos: chains and that multi-chain routing translates correctly to CAIP-27 chainId routing.

Tasks

  • Patch WalletConnectCommunicationClient to include both chains in session proposal
  • Extend wallet to handle WC2 session with two tezos: namespaces
  • Map operation_request.network → wc_sessionRequest.chainId
  • Extend headless mode to auto-approve WC2 session proposals (in addition to Beacon messages) — the wallet must call signClient.approve() automatically when HEADLESS=1. Note: WC2 proposals arrive asynchronously via signClient.on('session_proposal', ...); the wallet must initialize the WC2 SignClient, connect the relay WebSocket, and register the event listener before the test sends signClient.connect() — this is state-machine coordination that does not exist in the Phase 1–4 HTTP headless API and must be explicitly designed.
  • Validate with Reown relay — Project ID: fb4d4407a8fe167d79bd14b5afcc7230

Derisking questions

  • How should networks[] and version: "3" from permission_request be encoded in a WC2 session proposal? WC2 already has a multi-chain model (requiredNamespaces, optionalNamespaces, chains). The SDK patch must decide whether to map Beacon networks[] to WC2 chains, or carry the Beacon payload as opaque params — these lead to different capability detection mechanisms on the wallet side.
  • Design decision — gate avant Phase 5 : est-ce que networks[] et version: "3" sont mappés vers les requiredNamespaces/optionalNamespaces WC2, ou portés en opaque dans le payload Beacon à l'intérieur du message WC2 ? Ces deux options donnent des mécanismes de capability detection radicalement différents côté wallet (le premier couple le WC2 et le TZIP-10, le second rend le transport transparent mais impose que le wallet décrypte un blob). Cette décision doit être prise et documentée avant d'écrire du code Phase 5 — ce n'est pas une question de dérisking, c'est un choix d'architecture.
  • Un seul namespace tezos avec deux chains dans approved suffit-il, ou le wallet doit-il retourner deux entries séparées ? (impacte la structure de session.namespaces et la façon dont le dApp SDK lit les comptes)
  • Does the Reown relay accept a session with two chains in the same tezos namespace?
  • Does the WC2 protocol enforce account-per-chain in the session response?
  • Any rate limits or relay policy issues with a tezos: namespace for a non-mainnet chain?
  • Is a CAIP-25 profile update needed for the Tezos namespace to be accepted?
